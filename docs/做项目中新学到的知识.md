# 做项目中新学到的知识

## 1.什么是.world文件
Gazebo 的世界文件是 SDF 格式,本质是一个 XML 文件。它告诉 Gazebo：地面长什么样、光照怎么打、物理参数是什么、有哪些模型、各自摆在什么位置。

和 URDF 的区别：URDF 描述单个机器人，SDF 描述整个场景 + 场景里的一切。

<physics> — 物理引擎设置，有物理仿真步长，仿真速度，频率等
<light> — 光照
<model> — 物体
<visual name="visual">，给渲染用，决定物体长什么样
<collision name="collision">，给物理引擎用，决定碰撞形状
<link name="link">，把俩个包在一起
<static>，物体固定不动
<pose>，位置和朝向
<material>，颜色
<ambient>,环境光颜色
<diffuse>，漫反射颜色

写法：<box><size>1.2 0.6 0.03</size></box>类似这种把值包住

参数：
| x | 前后位置 | 米 |
| y | 左右位置 | 米 |
| z | 上下位置 | 米 |
| roll | 绕x轴旋转（翻滚） | 弧度 |
| pitch | 绕y轴旋转（俯仰） | 弧度 |
| yaw | 绕z轴旋转（偏航） | 弧度 |

## 4.解决slam建图时机器人穿过过高的物体问题
### 问题描述
在自定义 Gazebo 办公室世界中进行 SLAM 建图和自主导航时，TurtleBot3 waffle_pi 会从桌子下方穿过。原因是桌面高度较高，而 2D 激光雷达扫描高度较低，雷达只能看到桌腿，无法把整张桌子识别为完整障碍物，导致导航代价地图把桌子下方区域判断为可通行区域。
### 问题原因
Gazebo 中桌子的视觉模型正常显示，但其可被导航感知到的障碍信息不足。对于 TurtleBot3 waffle_pi 来说，2D 雷达主要用于 SLAM 和避障，如果桌子主体高于雷达扫描平面，机器人就可能在路径规划时从桌面下方穿过。
### 解决思路
不修改原有桌子模型的外观和位置，只在每张桌子下方添加不可见的碰撞体，使导航系统把桌子占据区域识别为障碍物。
### 解决过程
检查 .world 文件中的桌子模型，确认桌子由桌面和桌腿组成，桌面高度较高。
初步尝试给桌子添加低位薄碰撞板，但机器人仍可能从桌子下方穿过。
将薄碰撞板改为更稳妥的不可见实心碰撞盒，覆盖桌子桌面投影范围。
给每张桌子添加如下碰撞体：
<collision name='navigation_laser_blocker'>
  <pose>0 0 0.48 0 -0 0</pose>
  <geometry>
    <box>
      <size>1.55 0.85 0.92</size>
    </box>
  </geometry>
  <surface>
    <contact>
      <ode/>
    </contact>
  </surface>
  <max_contacts>10</max_contacts>
</collision>

### 补充说明
该方法只增加 collision，不增加 visual，因此不会改变桌子的显示效果，也不会影响后续相机对桌面目标物体的视觉识别。

<include file="$(find turtlebot3_bringup)/launch/turtlebot3_remote.launch">

<!-- Gazebo world -->
<include file="$(find gazebo_ros)/launch/empty_world.launch">
  <arg name="world_name" value="$(find office_robot_sim)/worlds/office.world"/>
  <arg name="paused" value="false"/>
  <arg name="use_sim_time" value="true"/>
  <arg name="gui" value="true"/>
  <arg name="headless" value="false"/>
  <arg name="debug" value="false"/>
</include>

同时最好加个全局时间：<param name="/use_sim_time" value="true"/>
/ 开头的斜杠表示全局参数。

## 6.解决机器人模型在RViz里看不到的问题
### 原因：
world 里自带的机器人是静态壳，没有 TF、没有电机控制器、没有 sensor 插件。导航栈收不到 /tf、/scan，RViz里就看不到机器人。
### 解决：
把机器人从 world 文件里删掉，改由 launch 文件动态生成——动态生成才会带控制器、LIDAR 插件、相机插件和 TF。

在world文件里删掉机器人模型：
搜索turtlebot3_waffle_pi，删掉整个<model name="turtlebot3_waffle_pi">...</model>块。

在launch文件里加：
<param name="robot_description" command="$(find xacro)/xacro '$(find turtlebot3_description)/urdf/turtlebot3_waffle_pi.urdf.xacro'"/>
  <node pkg="gazebo_ros" type="spawn_model" name="spawn_urdf" args="-urdf -model turtlebot3_waffle_pi -x -3 -y -4 -z 0 -param robot_description"/>
  <node pkg="robot_state_publisher" type="robot_state_publisher" name="robot_state_publisher">
  <param name="publish_frequency" type="double" value="50.0"/>
</node>

## 7.解决机器人在rviz里没有空降到特定位置的问题
### 原因
这是AMCL的正常行为AMCL，刚启动时不知道自己在哪，默认把机器人放在地图原点 (0,0)。
### 解决
第一步：在rviz里用2D Pose Estimate手动给一个初始位姿估计。
第二步：在 navigation.launch 里传参，让 AMCL 启动时就用你指定的初始位姿。
找到有关AMCL的，加入位置变量，并以</include>结尾，同时记得把第一行的<include .../> ，中的/删去，不然会说标签不匹配：
<arg name="initial_pose_x" value="-3.0"/>
<arg name="initial_pose_y" value="-4.0"/>
<arg name="initial_pose_a" value="0.0"/>

## 11.解决导航多目标点导航返回起点问题
### 问题一：到达目标点后震荡 ABORTED，日志报"无法到达"
现象：机器人走到目标点附近后原地来回微调，最终 move_base 报 Aborting because the robot appears to be oscillating over and over，状态变为 ABORTED。

排查：检查 dwa_local_planner_params_waffle_pi.yaml，发现 xy_goal_tolerance: 0.05（5cm）。TurtleBot3 在 Gazebo 中有 TF 噪声和物理漂移，5cm 精度 DWA 无法稳定达到，触发 move_base 的 oscillation_timeout: 10.0s 震荡检测。

修复：
xy_goal_tolerance: 0.05 → 0.20   # 20cm 容差
oscillation_timeout: 10.0 → 0.0  # 关闭震荡超时
oscillation_reset_dist: 0.05 → 0.10
controller_patience: 15.0 → 30.0
planner_patience: 5.0 → 20.0

原理：ROS Navigation 的 move_base 在目标点附近用局部规划器（DWA）做最终对准。xy_goal_tolerance 决定了"到达"的精度门槛。仿真环境中的里程计噪声和 TF 抖动使得 5cm 精度不现实，放宽到 20cm 后 DWA 能稳定收敛。关闭 oscillation_timeout 是因为震荡检测在松弛容差下已无必要。
### 问题二：全局代价地图膨胀半径过大，机器人"自锁"
现象：机器人到达目标点后发送回家目标，全局规划器报"无法到达"，但机器人物理上已在起点附近。

排查：inflation_radius: 1.0 意味着每个障碍物周围 1m 都是高代价区域。机器人到达目标后自身被膨胀障碍包围，全局规划器认为起点在障碍物内，拒绝规划。

修复：
inflation_radius: 1.0 → 0.30
cost_scaling_factor: 3.0 → 10.0
track_unknown_space: false  

原理：cost_scaling_factor 控制膨胀代价的衰减速率。值越大代价衰减越快，配合较小的 inflation_radius，障碍物只在实际几何边界附近产生代价，不会过度膨胀到阻塞通道。

### 问题三：TF 变换超时——costmap 启动时报错
现象：
Timed out waiting for transform from base_footprint to map

排查：global_costmap_params.yaml 中 transform_tolerance: 0.5（0.5 秒）。Gazebo + AMCL 启动初期，map→base_footprint 的 TF 树还未建立完整，0.5 秒的容忍窗口不足以应对初始化延迟。

修复：
transform_tolerance: 0.5 → 3.0   # 3 秒容忍

### 问题四：actionlib 状态机竞态——PREEMPTING/DONE 崩溃
现象：
Received comm state PREEMPTING when in simple state DONE
多目标点切换时 SimpleActionClient 内部状态机崩溃。

排查：py节点文件里的send_goal() 每次开头无条件调用 client.cancel_goal()。上一目标已进入 DONE 状态后，cancel 请求到达 action server，server 回 PREEMPTING 状态更新，但 client 内部状态机在 DONE 状态不接受 PREEMPTING 转换 → 竞态崩溃。

原理：ROS actionlib 的 SimpleActionClient 内部是一个有限状态机。send_goal 自身已处理旧目标取代逻辑，不需要外部显式 cancel。需要 cancel 的唯一场景是超时处理时的强制中断。

### 问题五：死胡同终点——机器人面对墙壁无法掉头
现象：机器人在最后一个目标点 (3, 3) 面朝墙壁/桌子，回家时 DWA 尝试跟随全局路径但前方是障碍物，机器人反复进退不转弯——"死胡同效应"。

排查：(3, 3) 是走廊尽头。全局规划器规划的路径从 (3, 3) 出发沿走廊返回，但 DWA 只能看到局部窗口，采样到的前向轨迹都撞墙。虽然后退轨迹理论上可行（min_vel_x: -0.10），但评分函数中 goal_distance_bias 占主导，面朝目标的旋转轨迹评分更高，导致"面壁思过"。

修复：两段式回家策略，在节点文件里先发一个中间点让机器人掉头，再发最终目标。

### 问题六：DWA 评分失衡——"面向障碍物"综合症
现象：即使有正确的全局路径，机器人在转弯点仍然"痴迷"面向障碍物，不停怼桌子，偶尔转到正确方向又"发癫"转回去。

排查：之前 occdist_scale: 0.08 太弱。当 goal_distance_bias 主导时，朝向目标但贴近障碍物的轨迹"看起来"总分最低。两个极小值相近时 DWA 在两个方向间反复横跳

修复：障碍物代价一票否决
occdist_scale: 0.08 → 1.0       # 12.5 倍放大
sim_time: 3.0 → 5.0             # 预判距离从 0.6m → 1.0m
goal_distance_bias: 20 → 32     # 适度目标牵引
path_distance_bias: 5  → 16     # 适度路径跟随
max_vel_x: 0.26 → 0.20          # 降速配合更远预判
max_scaling_factor: 0.2 → 0.5   # 近障自动降速更多
